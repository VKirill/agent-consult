# Решение проблем и Мониторинг (Troubleshooting & Observability)

Данный документ содержит инструкции по диагностике, отладке и мониторингу работы ИИ-агентов в составе консилиума «Агент Консалт».

---

## 1. Логирование реального времени (Observability)

Каждый запущенный локальный агент перенаправляет свой `stderr` в консоль родительского процесса с префиксом своей роли. Например:
```
[Агент: CLAUDE] Вызов инструмента: Read (аргументы: {"path": "prisma/schema.prisma"})
[Агент: CLAUDE] Инструмент Read вернул результат.
[Агент: CODEX] Resolving imports for index.ts...
```

*   **Шум спиннеров и Node.js предупреждения**: Встроенный фильтр в `src/consult-orchestrator.ts` автоматически удаляет из логов системные предупреждения (`ExperimentalWarning`, `DeprecationWarning`) и символы анимации прогресс-баров (`⠋⠙⠹`), оставляя только чистую отладочную информацию.
*   **Событийный лог для Claude**: Для Claude CLI включен стриминг в формате JSON (`stream-json`), что позволяет точно перехватывать вызовы инструментов МЦП и логировать их с высокой детализацией.

---

## 2. Динамический таймаут и Liveness Probe

Для борьбы с «зависаниями» агентов на тяжелых запросах реализована концепция **динамического пульса**:
1.  **Базовый таймаут**: Настраивается в `config.json` (`timeout_ms`, по умолчанию `240000` — 4 минуты).
2.  **Liveness-буфер (45 секунд)**: При получении любых данных от агента в `stdout` или `stderr` система проверяет оставшееся время до принудительного убийства. Если осталось менее 45 секунд, дедлайн сдвигается вперед, гарантируя процессу еще 45 секунд работы.
3.  **Максимальный лимит (15 минут)**: Суммарное время работы одного агента ограничено 15 минутами (`900000 мс`), чтобы предотвратить бесконечные циклы вызовов инструментов при багах.

---

## 3. Очистка ресурсов и дочерние процессы (Zombie Processes)

Локальные агенты запускаются в изолированных группах процессов:
*   **Конфигурация**: `spawn` выполняется с параметром `{ detached: true }`.
*   **Убийство процессов**: При срабатывании таймаута или ошибке, вызов `killProcessGroup()` отправляет сигнал всей группе процессов (отрицательный PID на Unix/Linux: `process.kill(-child.pid, 'SIGKILL')`).
*   **В среде Windows**: Для надежного завершения дерева процессов вызывается системная утилита: `taskkill /pid PID /f /t`.

Это предотвращает накопление в системе зависших процессов `git`, `npm` или компиляторов, порожденных агентами.
