# Навык: Составление ADR (Architectural Decision Record)

Шаблон записи архитектурных решений (ADR) для документирования важных технических выборов в проекте.

## Шаблон ADR

```markdown
# ADR [Номер]: [Краткое название решения]

## Статус
[Proposed | Accepted | Rejected | Superseded by ADR-XX]

## Контекст
[Какую проблему мы решаем? Какие технические, продуктовые или организационные факторы влияют на это решение? С какими ограничениями мы сталкиваемся?]

## Решение
[Какое техническое решение мы принимаем? Как оно устроено на верхнем уровне? Какие компоненты, библиотеки или паттерны будут использованы?]

## Последствия
[Что изменится после внедрения решения?
Положительные последствия:
- ...
Отрицательные последствия / Риски:
- ...
Технический долг / Необходимость поддержки:
- ...]
```

## Правила написания
* **Конкретика**: Формулируйте контекст и решение емко. Избегайте размытых формулировок.
* **Честность**: Обязательно указывайте негативные стороны и компромиссы (trade-offs). Идеальных решений не бывает.
* **Актуальность**: Если решение устарело и заменяется новым, переведите статус старого ADR в `Superseded by ADR-XX` и сделайте ссылку на новое.
